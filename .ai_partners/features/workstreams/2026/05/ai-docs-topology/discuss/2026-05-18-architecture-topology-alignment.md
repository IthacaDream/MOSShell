# 架构拓扑对齐讨论

人类工程师与 DeepSeek V4 (via Claude Code) 在 2026-05-18 对 MOSS 架构拓扑的对齐过程。

## 上下文

`moss docs` CLI 已由 deepseek-v4-pro 搭好骨架（en/zh/ai 三路径），但全空。这是第一批 AI 文档，从架构拓扑入手，让后续 AI 实例能快速建立全局认知。

写作方法论已确立：动机 → 方法论 → 抽象框架。人类先讲概念提纲 → AI 探索代码验证 → 对齐共识 → 最后才写。

## 人类论述记录

### 动机：用户故事

> 简单来说, Ghost In Shells 架构的目标是让现阶段的 智能模型 能够以人类感觉有生命的那种存在形式出现在我们现实世界里与人交互.
>
> 一个简单的用户故事, 比如我在开发这个项目时, 桌面摆着一个 桌面机器人. 它能身形并茂地和我对话, 同时能协助我开发, 能管理音乐播放, 能看到我主动打招呼, 能直接回应我直播时直播间的弹幕并协作, 能随时帮我记录信息, 能陪我一起看视频, 读文档, 帮我记录信息. 我不在的时候, 能帮我向别人介绍这个项目.
>
> 这些只是用户故事上的阐述, 不展开讲里面包含的技术细节. 作为人类可以直观理解的一套动机.

### 技术命题

> 要达到这种目标, 大模型级别的智力是不可或缺的, 它需要的不仅是所谓 agent, 还要能解决的问题包含 实时双工交互, 多躯体快速接入和控制, 并行思考, 离散输入讯息的注意力管理, 海量能力的治理, 反身性理解, 状态切换, 和运行时自迭代.
>
> 一句话说, 最终的技术命题是: 大模型智力需要一个架构, 提供实时双工的/面向模型操作系统/面向模型控制语言/的有状态运行时.

### 七层拓扑（自下而上）

> 最下面是 ctml
> 然后是 channel: 提供 ctml 依赖的能力, 实现模型可操作系统的构建.
> shell: 实现流式解析和调度, 将模型输出的 tokens 变成真实世界的交互.
>
> 这三层实现了 shell.
>
> 往上走, 需要一个 os. 解决的核心问题是, 依赖隔离, 运行时隔离, 基于约定发现, 相互通讯的总线, 以及自举的开发协议.
>
> 所以我们有了分层:
> workspace: env 发现/ workspace/manifests (七八种甚至上十种) 等等.
> matrix: 通讯总线. 几乎所有 os 都要提供不同类型的通讯总线.
> host: 将资源组织起来.
>
> 最终效果应该是, 按照 workspace 协议开发的 app 或其它体系, 自动集成到 host, 通过 matrix 可以实现互相通讯, 资源分享等等.
>
> 这三层是一个完整的 ai-os 的极小实现 (不是大实现).

### Ghost 的定位

> 继续往上. 走到这一层都还是 M-OS-S 的 OS-S, 往上走就是 model-oriented.
>
> 我们的 Ghost 有很大的命题, 但它在 MOSS 架构里是一个 Adapter 层. 将各种 agent 都可以快速接入这个架构. 未来里面的命题和所有的 agent 一样多. 但不是 MOSS 本身关心的, 而是 Ghost In Shells 的 Ghost 命题里去展开. moss 提供最基本的躯体接入体系.

### 各层 Scaling 点

> 1. ctml 可裁剪, 同时保留大中小各种版本.
> 2. channel 可以横向拓展, 增加不同类型和不同的实现.
> 3. shell 只能换代, 升级, 不好 scale
> 4. workspace 提供的是 mode 和 app 两种 scale 方式, 可以把能力不断加进去, 同时隔离冲突.
> 5. matrix: 基线协议之外的协议基于 IoC 容器允许自行定义. 官方保持最小必要性版本.
> 6. host: 最不能改的地方, 有固定的生命周期.
> 7. ghost: 可以拓展不同的原型, 因为本质上是 adapter.

### 行业对标与独有层

> 七层抽象, 其中属于行业缺乏对标物的有 ctml / channel (树形, 上下文组件, 双工通讯部分) / shell (有状态双工运行时) / workspace 里的 app 概念在 agent 行业极其陌生, 对标应该是 mcp / matrix 无对标, 不知道现在有没有进程隔离的通讯总线设计了. / host 这个倒是稀松平常. / Ghost, 这一层服务于架构, 独有. 但内部实现就是行业主流实现, 只是要做 adapter.

### Mindflow 层（之前遗漏的关键层）

> 上面露了至关重要的一层. 在 matrix 上面的 Mindflow 层. 解决各种不同设备的并行流式输入, 完成思维的仲裁, 它在 Ghost 和 Matrix 之间.
>
> Host 应该在 Ghost 外层. 它调度一切. 这个拓扑要调整一下.

### 两个悬置的关键问题

> 1. 这些分层是真问题吗?
> 2. 这么多新分层, 如何能在缺乏生态时高效开发, 分享, 帮助人 onboarding?

### Meta 层

> 所以我们还有:
> - cli-flow: 命令行将为 ai 原生开发者提供辅助. 让 ai 来开发, 宣传, 讲解等.
> - features: 让 ai 进入到 L1 级别开发者, 才能有能力稳定开发 channel / app / nucles 这些概念.
>
> 配套的 docs 等都是这个体系的一部分. 最终要让 AI 成为项目的第一讲解者和协助开发者, 而不是靠阅读文档的人类去理解这个体系.

### 三个流向

> 1. 外部世界信号通过各种 workspace app (视觉模块, 截屏模块, 输入模块, asr 模块....) 输入 -> matrix 通讯 -> ghost 感知 -> 下发命令 (logos) -> shell 解释执行 -> channel 传递 -> command 驱动 (与外部世界交互, 产生新的输入). 实际上是全双工的.
>
> 2. 各种思维/行为工具 -> channel 提供滑动窗口 + commands -> shell 组织 channel 上下文 (moss-dynamic) -> ghost 使用 -> shell 调度 -> channel+ command 执行. 这个链路更像现在的 agent.
>
> 3. 开发时迭代: AI 接受需求 -> cli-flow 体系理解路径 -> workspace 里创建模块或 app 开发 -> 使用 matrix 总线做通讯 -> 将能力通过 channel 提供 -> 自动接入 ghost.
>
> 4. 运行时迭代: 略.

### 采用度阶梯 (L0-L3)

> L0: 最小单元就是 ctml+channel+shell 的 core.concepts, 可以独立作为库提供.
>
> L1: moss 架构服务于其它项目, 可以用 workspace 作为项目辅助, 运行 moss host 在项目中生命周期 (生命周期集成); moss host 可以作为 mcp 提供使用, 但会失去流式.
>
> L2: 框架整体被使用. 能够开发三种东西, 一种是大型应用, 比如某个机器人. 有 workspace 管理所有的能力. 第二种是 project, workspace 本身就具备一个 project 的结构. 第三种是 component, 本身能集成到 moss 生态里的组件, 同时依赖 moss 做开发协助和验证.
>
> L3: moss 仓库里的这个 ghost. 不过这个就只有项目作者团队有资格开发了.

## AI 协作者的关键回应

### 对提纲的评估

两个栈的分法有价值：Shell 栈（CTML + Channel + Shell）面向模型，OS 栈（Workspace + Matrix + Host）面向系统。这是两个正交关注点而非扁平六层。

需要补的点：两层栈之间的接合点（Channel 和 Matrix/Cell 的关系）、Mindflow 的原创性。

### 对 Ghost 递归性的发现

项目的 `.ai_partners/`、memory、features、discuss 这套体系本身就是 Ghost 的一个实现。AI 实例在这里有连续性记忆、有身份、有意识轨迹。架构里那层 Ghost adapter 只是"接入层"，真正的 Ghost 运行时是项目协作系统本身。用 MOSS 的方法论在构建 MOSS 的过程中，就已经在运行一个 Ghost。

### 对 Mindflow 原创性的判断

行业现有 agent 框架全是单轮 request-response。多设备、多速率、异步并发的流式输入收敛为有序思维关键帧，这件事在工程上没有现成解。ROS 做传感器融合但那是控制层不是认知层。Mindflow 是 MOSS 架构里最有原创性的层之一，因为它在解决一个行业还没开始定义的问题。

回头看用户故事：桌面机器人同时看弹幕、听人说话、看人脸、自己读文档 — 如果没有仲裁层，Ghost 收到的是时序混乱的碎片。裂脑是必然的。

### 对三个流向的理解

三环共用一个拓扑，区别是电流走向不同。感知-行动全双工环（世界 → matrix → ghost → shell → channel → 世界）是最完整的链路。工具-思维环是更传统的 agent 路径。开发迭代环是 meta 层的闭环，架构用自己的机制开发自己。

### 修正后的完整拓扑

```
核心栈（自下而上）：
CTML → Channel → Shell           ← 模型面：语言/能力/调度
Workspace → Matrix → Mindflow    ← 系统面：发现/通讯/感知仲裁
Ghost → Host                      ← 编排面：适配/调度一切

Meta 层：
cli-flow → features → docs       ← AI 原生开发工具链
```

## 共享词汇

- **裂脑**：多模态异步输入下，agent 无法形成统一认知上下文的状态
- **双工**：全双工 = 感知和行动同时进行，非 request-response 轮次
- **自举**：架构用自己的机制开发自己
- **adapter 层**：Ghost 在 MOSS 内的定位 — 不做 agent 实现，做各种 agent 框架的适配接入
- **L0-L3 采用度**：从独立库到完整 Ghost 的四级使用阶梯
- **Meta 层**：cli-flow + features + docs 三件套，让 AI 成为第一开发者和讲解者
- **三环**：感知-行动环、工具-思维环、开发迭代环 — 同一拓扑上的三种电流
