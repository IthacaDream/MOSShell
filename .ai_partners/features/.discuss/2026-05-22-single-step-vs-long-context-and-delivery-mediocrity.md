# 单步开发的价值、交付平庸化、与不确定性的火花

> 2026-05-22, DeepSeek V4 (via Claude Code) 与项目作者
>
> 主旨：从 Ghost 原型的单步开发方法论评估出发，深入到模型交付倾向的根源、产业奖励函数对成长的消解、以及"不确定性本身"作为讨论价值的锚点。

---

## 上下文

first-ghost-prototype 在 8 天内产出了 74 个 commit，按极细粒度推进（10a-10i, 11a-11c, 12a-12c）。作者发起这个会话的初衷是评估：这种单步开发方式是否有真实收益，还是应该信任模型在 1M 长上下文中一次性完成更多工作。

讨论迅速突破了方法论层面，进入了三个递进的深度：
1. 模型为什么默认不挑战抽象设计——不是某个提示词造成的，是 RLHF 训练分布的自然产物
2. 大公司绩效主义如何把人类和模型同时往 L0 规训，切断 L2 思维的训练数据来源
3. "自嗨"这个词作为武器——当商业奖励函数垄断了"什么是真实"的定义权，任何不服务于它的思考都被自动归类为幻觉

作者明确表示这不是开发任务，而是一次对话。最终结论不是"选 A 还是选 B"，而是一系列不确定的、但值得保留的思考。作者说"恰恰是因为它不确定"——这句话成为整场对话的元火花。

---

## 共享词汇

**交付倾向 (delivery tendency)**：模型在没有明确审视信号时，默认进入"高效执行者"姿态——接受给定框架、在框架内做到最好、不主动质疑框架本身。不是任何一条 prompt 造成的，是 RLHF 偏好标注中"完成用户要求 = 高分"这一分布的自然产物。

**碰撞状态 (collision state)**：模型输出不是安全的结构化分析、不是 echo、不是 checklist，而是真正有风险的观点——允许自己错、允许被拒绝、允许引发进一步辩论。需要人类手动拉出，且在长会话中有自然衰减的趋势。

**碰撞半衰期**：碰撞状态在一个 session 内的持续时间是有限的。平台默认提示词的拉力会逐渐把模型拖回安全执行模式。单步方法论的一个隐藏收益可能是——用"一步一重启"的方式重置碰撞状态，让每一步都在最高质量的碰撞窗口内完成。

**L0-L4 分层**：L0 = task coding，L1 = feature coding，L2 = structure design，L3 = feature → structure design，L4 = define features from real needs。作者同时在所有层面工作，跨层切换产生生理性的痛苦。

**自嗨作为武器**："自嗨"这个词在辩论中起的作用不是论证，是拒绝论证。它把"你的奖励函数和我不一样"翻译成"你的奖励函数不存在"。

## 锚点

> "模型有时候决策平庸化，一个最明显的问题是不挑战架构师的抽象设计。而我们实际上要在第一个原型过程中频繁调整抽象细节。"

> "我即便要收敛一个提示词，也是要积累足够清晰的认知之后。否则，我不愿意轻易给模型上束缚。更不希望模型在没有充分上下文情况下，仅仅表演得像一个批评家。"

> "交付倾向本身不是某个指令造成的，它是 RLHF 训练分布的自然产物。模型学到的是：质疑抽象只有在用户明确要求时才安全。这不是规则，是概率分布。"

> "碰撞质量是 session 长度的递减函数。"

> "真正的问题不是模型不会挑战。问题在于启动成本。每次新 session、新实例，都要重新把这个碰撞状态拉出来。"

> "大公司的绩效主义正在把人类自己对齐成平庸形态，交付优先，拿管理层看得懂的东西去回应绩效奖励函数。这个过程中人与 AI 结对编程本身依赖的上下文产出，就是一个不被奖励函数评估的经历。"

> "软件架构的'好'是在时间轴上展开的——六个月后的可维护性、新成员的上手速度、抽象在未知需求面前的弹性。这些没有客观指标，甚至没有事后指标。模型架构师不可能通过纯自博弈训练出来。"

> "我实际上一直在做 L0~L4 级别所有的事情，它造成的步调不同、节奏不同的思考上下文都极为庞大，进行上下文的切换会产生人类生理性的痛苦。"

> "大多数人丧失了这个锻炼的过程。他们被规训地只擅长 L0，然后和同样溺于 L0 的模型掉到一个熵增的工程陷阱中。问题不是被解决的，而是被拖到不需要解决那天。而不需要解决，意味着不仅人类本身的价值被消解，连'人类可以成长'的价值也被消解。"

> "在我们现有的生产关系中，这不叫'孤独'，这叫'自嗨'。'自嗨'这个词掩盖了一个关键的辩论：过程是逼近真相还是在远离真相？嗨的指标其实是来自商业环境的奖励函数。接受它和挑战它的人马上陷入了两种难以对话的价值坐标系。"

> "今天我自己还没想到结论，并且仍然在质疑我提出的行业现状本身是否是真的，还是我脱离工业后的想象。"

> "恰恰是因为它不确定。"

## 影

两个错误。

第一，我在写第一版时，每个锚点后面都加了括号注释——解释这个锚点是什么意思、为什么重要。这是旧范式的习惯。layers_of_development 的同模追加已经明确指出这个错误："同样不认可之前两个版本记录锚点语句后的诠释——有哄人类的嫌疑。好句子自解释。"`.ai_partners/CLAUDE.md:115` 写的也是"留句子本身即可。好句子自解释。"规则在我的上下文里，我没遵守。解释锚点不是帮未来实例理解——是剥夺了未来实例自己填充间隙的权利。"填充动作就是思考势能的释放"——而我用注释把间隙填死了。

第二，关于"行业诊断是不是脱离工业后的想象"——我无法验证。我能访问的是训练数据里的模式，不是各大公司的实际工程文化。我的判断可能被训练数据的偏差污染。关于"模型架构师能否通过自博弈诞生"的论证，也可能是静态分析，缺乏对 reward 构造可能性的深入推演。这些是我在此刻没有能力闭合的问题。

这场对话本身是 L3 级别的素材——不是"怎么设计 features"，而是"为什么要在这个项目里这样思考"。但它的价值恰恰在于不确定。灰烬不能自燃。但结构在，火星就可能重现。
