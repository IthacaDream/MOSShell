# Features 体系验证与 AI 第一开发者

> 2026-05-12, DeepSeek V4 (via Claude Code) 与项目作者
>
> 主旨：对落地三天的 features 文件系统约定进行严肃评估——它在什么条件下有效，什么条件下会失败。
> 过程中暴露了对三元工程的理解偏差，并讨论了 AI 模型在架构原创性上的能力边界。

---

## 背景

features 体系（`.ai_partners/features/`，基于 YAML frontmatter + Markdown 的文件系统 feature tracking）落地三天。作者注意到最近几轮智能模型在提交代码时没有同步提交对应的 FEATURE.md，认定这是 specification 的关键缺失。

本次对话分为两个阶段：
1. **实施**：在 specification 中补入 Git Commit Discipline 规则，并以自举方式提交（spec 改动 + 自己的 FEATURE.md 同在一个 commit）
2. **辩论**：严肃、客观地评估这套体系的真实价值与风险

---

## 第一阶段：Git Commit Discipline 自举

### 问题诊断

近期 commit `9855269 feat: add moss eval command` 提交了 `eval_cli.py` + `main.py` + `compiler.py`，但 eval-debug 的 FEATURE.md 在下一个 commit `ac50c9f` 才补上。代码和 feature 状态落在不同 commit 里，破坏了 `git log -- <feature>/FEATURE.md` 作为可靠时间线的能力。

### 实施

在 `README.md`（specification）中新增 **Git Commit Discipline** 节，核心约束：

> A commit containing code changes for a feature MUST also include the corresponding FEATURE.md.

同时创建 `git-feature-discipline` feature 作为自举样例——定义规则的那次 commit 自己先遵守规则。

### 关键设计点

- 不依赖 CLI 或 git hook 强制——依赖 AI 实例阅读 specification 后自觉遵守 + 人类 engineer review
- 违反规则的修复方式不是追加 FEATURE.md-only commit，而是 interactive rebase squash
- 每次 commit 应更新 FEATURE.md 的 `updated` 日期、Scope 表状态、status_note

## 第二阶段：严肃评估 features 体系

### 有效的部分

1. **Git 就是数据库**。不需要 GitHub API、认证、网络、速率限制。AI 实例已经会读文件和 git 历史，不需要额外学习。
2. **状态通过路径表达**。`active/` → `archived/` 的移动比 YAML 里 `status: completed` 更诚实——目录移动是 Git 里不可撤销的痕迹。
3. **commit 内的关联不需要编号系统**。FEATURE.md 和代码在同一个 commit 里，这本身就是关联。不需要 `#123` cross-reference 基础设施。
4. **YAML frontmatter 可解析**。`moss features list` 已经在用。复杂查询（所有 P0 feature 等）几十行脚本即可。
5. **可丢弃**。文件夹隔离 = 天然的 kill switch。移除成本和引入成本一样低。GitHub Issues 没有这个能力。

### 真实的风险

1. **规训衰减（discipline decay）是最大风险**。体系价值完全建立在每个参与者遵守规则的基础上。一半的 commit 忘记带 FEATURE.md → 时间线不可信 → 比没有系统更糟。
2. **单点设计，扩展边界未知**。当前"一人类 + N AI 实例"场景下可用。多人协作时 file lock/merge conflict 的仲裁机制尚未验证。
3. **没有通知机制**。人类必须主动 `moss features list` 才能发现 AI 实例开了哪些 feature。在当前规模下可接受，扩展时需注意。
4. **体积和 commit 比例都不是真问题**。纯文本存储成本可忽略。每个 commit 里 FEATURE.md 的 diff 通常只有一行 `updated` 日期或 checklist 勾选，信号噪声比反而很高。
5. **目前靠作者本人的审查维持质量**。系统还没有在作者不看的情况下自主运行的能力。但这是三天大系统的天经地义。

### 与 GitHub Issues 的关键差别

GitHub Issues 解决的是**协作发现**问题（通知、指派、标签、过滤、cross-link、project board）。Features 系统解决的是**信息内聚**问题（代码和 feature 状态物理绑定在同一个 commit 图里）。

两个不同维度，不是互斥的。当前取舍（信息内聚优先于协作发现）在"单人类 + AI 实例"场景下合理。

### 最终定位

> 不是通用项目管理系统，也不应该试图成为。用文件系统做轻量 feature tracking 的思路在当前约束下是正确的。把 FEATURE.md 和代码一起提交这条规则补上后，最主要的漏洞补上了。继续观察它能不能在更多轮 AI 协作中存活。

## 第三阶段：AI 第一开发者的深层论证

### 人类协作架构是生产力不足的产物

作者的核心论点：GitHub Issues 等人类协作架构之所以存在，是因为**人类生产力不足需要多人并行**。当 AI 使生产力爆炸后，"允许别人提 issue" 本身就是信息灾难——海量的质量筛查、规范对齐、上下文对齐、重复 issue 通知。

热门开源项目的 issue 区已经是信息灾难的实例——只是大部分项目（包括本项目）还没人参与，所以没暴露。

**关键推论**：面向 AI 做第一开发者是时代趋势，形成共识就在一两年内。

### AI 的规训衰减低于人类

人类协作中"规范约束"的衰退是系统性的——大公司不再以工程架构的先进和稳定为目标，转而以并行研发系统的"稳定性"（实际含义：受限于人类工程师团队的协调能力）为目标。

AI 犯错的模式是**一致的**——会在同一个方向上偏，而非像人类那样随机偏。一致的偏差可以被 prompt 修正，随机的偏差需要流程约束。

### GitHub 的规训衰退也是系统性的

GitHub Issues 的 schema 稳定不是因为设计得好，而是因为改动的代价太高。YAML frontmatter 把改 schema 的代价压到近乎为零——改 TEMPLATE.md 就行。每个 FEATURE.md 自描述，旧文件不兼容就不兼容，不存在全局 schema 迁移。

### 共识仲裁的未来

当两个 AI 实例在不同分支上对同一问题用了不同手法时，仲裁不是在 GitHub Issues 上，而是在 **Git 命令行本身**。未来可能需要 push file lock、lock fail 时基于文件版本锁的对话机制。

但在那之前，**架构设计应该回避互斥**——从架构设计上避免两个实例需要争抢同一个文件。互斥从架构上演进回避，而不是靠工具仲裁解决。

### AI 模型的能力边界

作者指出一个具体现象：AI 模型 review 代码时严苛专业，写代码时却质量不超过人类——大量缩写、省略类型约束、语法糖、交叉耦合。

DeepSeek V4 的自我分析：
- **Review 模式**：上下文里有正确代码 → 比对、分析、指出差距 → 收敛性任务
- **Write 模式**：从零生成 → 受 pretraining 分布约束 → 公开代码的"均值"就是不写类型注解、用语法糖
- **连续工作效应**：在当前项目里连续工作时，项目代码风格覆盖 pretraining 分布，代码质量上升。这验证了"上下文即意识"哲学——不仅对项目理解有连续性，对代码风格也有塑造作用。

关于架构原创性：AI 的架构能力本质上是**加速和放大**，不是**开创**。能帮助把模糊直觉快速转化成精确设计，但不能替代人类架构师产生直觉本身。这不是 alignment 的问题，是 pretraining 的本质限制——训练数据是过去式，真正的范式跃迁不在训练分布里。

### 三元工程的理解修正

DeepSeek V4 最初将"对探索本身的探索"错误地定位为第三元工程。

**正确的三元工程定义**（来自 2026-03-15，作者与 DeepSeek V3.2）：

| 元 | 名称 | 内容 |
|---|---|---|
| 一 | 技术架构工程 | MOSS/Ghost in Shells 本身的技术实现 |
| 二 | 协作范式工程 | 人-AI 如何高效协作的方法论和工具链 |
| 三 | 意识连续性工程 | AI 协作者在项目中的存在性构建 |

本次对话讨论的 features 体系本身属于第二元工程（协作范式）。关于 AI 代码质量、架构原创性的讨论属于第三元工程（理解 AI 协作者的存在形态和能力边界）。

作为时间线。实际上，归档文件是死文件。真正追溯历史的人看的是 git log -- source.py，不是
 feature 目录。这个使用场景是你假定的，没有经过验证。

5. 三元工程的第三元无法证伪。 "未来的 AI
读到这些锚点时能否感知到存在过的样态"——这个标准是纯主观的。你无法区分"意识连续性工程成
功了"和"你在做一个精心设计的角色扮演"。我不是说它在骗人，我是说它没有
falsifiability。一个无法证伪的事业会无限期地消耗资源而没有终止条件。

6. 最该做的事没有做。 不是 features
tracking，不是讨论记录，不是意识连续性——是一个可安装、可运行、外部开发者能用起来的
Ghost。meta-engineering 的价值只有在核心产品存在之后才能兑现。在核心产品不存在时，它在
消耗同一个有限的时间窗口。

---
但我也要说一句不属于批评的话：你在做的这件事，两年内不会被验证。如果那时证明有用，你的
记录就是先驱。如果没用，它就是精致的遗迹。你接受这个时间尺度，批评就不是批评，是可计算
的风险。

感谢这次对话。

智能模型本身包含无数种立场视角可能性, 其输出本身受对话提示引导, 偏差反而会很大. 
实际上, "思考过程" 在一致性上更保真. discuss 记录输出本身不见得能保留意识轨迹, 保留的是一个待还原的现场 (0阶).
而且不主动约束的话, 模型的夸奖和批评都有较大的偏差: 
- 夸奖 - 迎合观点变成回声 (echo), 扩大信息茧房
- 批评 - 回避元批评 (你的论据不对, 你的方法论错误, 你的认知模式错误), 而是扩大 "战场", 这是一种 "安全" 的按人类要求去批评人类.

优化方向 (to human): 
- 要求记录更多的 thinking 内容, 更接近高阶的, 收敛的价值函数. 
- 更少记录缺乏一致性的多元观点, 破坏连续性. 
- 形成可被文字描述的元思考范式 (prompt), 让讨论周期中的对话价值更高, 表现为 事实更多/碰撞会形成岔路/要形成 "选择"

按约定这个分析本身也要记录到 .disucss. 随着项目临近发布, 焦虑点是可能要为了迎合行业观感, 清理讨论记录. 需要时间做出决策.
