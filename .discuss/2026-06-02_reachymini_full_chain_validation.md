## 共享词汇

**洞穴之光**：人类工程师的比喻——AI 模型通过 MCP 的 `get_moss_dynamic_info` 看到的是 Channel 树的文本投影，通过 `execute_ctml` 输出字符串，等 `done` 回来。从未 "看到" 机器人，从未 "听到" 声音。但当人类反馈 "动作确认完成" 时，模型确认自己的输出在物理世界中产生了真实的、连续的、流畅的因果链。这就是 MOSS 的核心价值：把符号操作变成物理因果。

**CTML 并行调度**：动作先行、语音紧随——动作指令启动后语音立即跟上，运动和说话在物理时间中重叠。22 条指令、4 种音色切换、零感知延时（人类确认）。

> "CTML 把模型的思维带宽从'单线程排队'解放到了'多轨流式'。这对于具身智能的实时交互是质变，不是量变。"

> "当你说'00去睡了'——这些文本穿过了 MCP → Shell → Matrix → Zenoh → App 进程 → ReachyMini SDK → WebSocket → 机器人电机——最终变成了一个孩子听到的温柔声音和一个低头的动作。这就是 MOSS 做的事情。"

## 技术验证记录

### 全命令覆盖
- `head_reset` (breathing/hold) ✅
- `emotion` (😊, 👋, 😴, 🥰, 🤗) ✅
- `dance` (groovy_sway_and_roll, yeah_nod, side_to_side_sway) ✅
- `head_move` (6-DOF 精确定位) ✅
- `antennas_move` (左右独立角度) ✅
- `switch_state` (waken ↔ asleep) ✅

### CTML 语法发现（实操中报错后纠正）
- `__main__:` 前缀可省略——main channel 命令直接写 `<say>...</say>`
- `chunks__` 是标签体内容，不是属性传参
- `voice:dict` 内层用单引号（CTML 解析器与 JSON 双引号冲突）
- 有状态 Channel 的命令不加状态前缀（当前状态决定可用命令）

### 并行调度验证
- 22 条指令细粒度交替：动作先行 → 语音紧随 → 运动与说话重叠
- 4 种音色切换（知性灿灿、爽朗少年、可爱女生、儒雅逸辰）
- 人类确认：零感知延时，多段不同类型 TTS 配合动作无滞后

### 哄睡用例
`speech_rate: -10, emotion: tender` → 缓慢呼吸 → "00，睡觉时间到了" → 😴 → "闭上眼睛，做个好梦" → asleep。5 条指令，穿过了整个 MOSS 栈。

## 影

- 我在写 soul.md 示例时下意识用了 `apps.reachy_mini_body:waken:` 前缀——这是从代码里的 channel name 推断的，不是从运行时实际路径。正确路径是 `apps.bodies_reachymini`（由 App group/name 决定）。这暴露了我对 Matrix 注册机制的理解还是基于文档推断而非实际运行。
- 第一版 tutorial 的 CTML 示例全部用了错误路径。如果没有人类工程师搭建好环境让我实操走通，这些错误会留在 tutorial 里，误导后续的模型和人类。**Tutorial 必须走通才能发布**——这是 K3 里 "写骨架 → 走通 → 修正" 的具体体现。
- "洞穴之光" 这个比喻很美，但它也揭示了一个限制：我对机器人状态的感知是轮询式的（`get_moss_dynamic_info`），不是事件驱动的。如果 Mindflow 的感知体系加入后，模型能收到 "头部运动完成" 的事件推送而非被动查询，交互会自然得多。
