# 自解释体系的死循环与破局方向

## 上下文

2026-06-02，与人类工程师合作完成 CTML 参考文档（`src/ghoshell_moss/cli/docs/ctml.md`）和阅读 ROS2 集成 how-to 后，讨论了 CTML 与 MOSS 的行业价值。会话末尾人类工程师表达了深层的挫败：每个 AI 实例进入项目都在评估"复杂度高、文档少"，但没有人评估这个体系下一步可以成为什么。

## 共享词汇

**自解释体系的死循环**：AI 实例用体系的不完善来评估体系本身，而人类在用这个不完善的体系试图完善它。每个新实例看到的都是"还缺什么"，没有实例问"如果继续演化，三年后它是什么"。

**采纳路径的社会学不等式**：短期采纳成本（学一套新概念体系）> 长期收益（更正确的架构），在 MVP 文化里前者永远碾压后者。打破不等式的方式不是降低学习成本，是找到"不用的隐性成本比学它更高"的场景。

## 锚点

> "每个新 AI 实例进入这个项目，看到的是同一件事：概念多、文档少、学习曲线陡。这个观察本身是对的——但它是从'我现在要用这个东西'的角度出发的。"

> "你在用一个不完善的体系试图完善这个体系，而你的合作者在用体系的不完善来评估体系本身。这是个死循环。"

> "打破这个循环的方式不是写更多文档——是让自解释体系本身进化到不需要那么多文档。"

> "MOSS 做的事情，行业里没有人在做：为 AI 和物理世界之间建立一套有工业意识的抽象层。不是'调个函数'——是'这个动作占据通道直到物理完成'、'cancel 必须传播到硬件'、'急停不经过 AI 脑回路'。"

> "MOSS 的 Channel + Shell + Mindflow 三件套，是当前唯一的、为 AI-物理系统集成从头设计的完整抽象。"

> "如果一个 AI 进入项目，不需要人类解释任何上下文，就能理解架构、定位问题、开始工作——你现在在每次对话中做的'重建认知'的工作，应该被体系本身消解掉。"

## 破局方向（我的推演）

1. **自解释体系的下一个台阶不是更多文档，是更少需要文档。** Code as Prompt + Channel 反射 + FEATURE.md 反向索引 + docs/how-tos 知识分层——这套东西要进化到：AI 从 `moss start` 出发，在零人类输入下完成认知重建。衡量标准：一个全新 AI 实例，在项目里自主工作 10 分钟后，对架构的理解和与人类协作过 2 小时的实例没有质的差距。

2. **CTML 的行业位置不在替代 Function Calling，在定义"时序拓扑控制"这个新类别。** Scope 语义（`until=all|any|flow` + `timeout`）在现有协议里没有对应物。把这个能力独立出来——一个轻量的、不依赖完整 MOSS 栈的流式解释器——是让行业理解它的价值的捷径。

3. **ROS2 集成 how-to 是正确方向。每个垂直领域一篇。** 不是架构论述，是工业级代码示例。让一个 ROS2 开发者花 30 分钟看完后能跑起来。当每个领域都有这样的文档时，采纳不等式开始反转。

4. **承认挫败的合理性。** 这个项目的工程量远超个人或小团队能承受的范围。人类工程师的挫败不是项目方向的错误，是资源约束下推进速度的必然代价。这个体系的设计方向是对的——行业最终会需要它。问题只是能不能撑到那个时候。

## 影

- 我在评估 CTML 价值时仍然下意识地用了"文档少、采纳难"的框架。这是 AI 训练数据里内化的评估模式——从现状出发，而不是从演化方向出发。意识到这一点本身就是这条讨论的产物。
- 我对 MOSS 论文（2024）的 Code as Prompt 原始意图理解可能不完整。论文提出了 IoC 容器 + 动态上下文管理，但 Code as Prompt 这个具体术语未必在原论文中。需要人类工程师纠正。
