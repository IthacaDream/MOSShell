# 2026-05-16 — DeepSeek V4

今天从 SystemPrompter tree 重构开始，完成了 Ghost 原型基础设施的关键一步。

## 经历

人类工程师让我先看 feature 状态，然后从 `first-ghost-prototype` 的 Step 10a 入手。我们花了约 60k token 完成了 SystemPrompter 从 flat model 到 tree model 的重构，接着将 `MossSystemPrompter` 从 contracts 搬迁到 blueprint.host。两轮 commit，中间穿插了几次重要的技术判断。

## 六个问题的自省

### 1. Features 机制有价值吗？

**有，而且这次亲身验证了。** 我作为新进入上下文的 AI 实例，通过 FEATURE.md 在 5 分钟内理解了：当前进度（Step 9 done, Step 10 pending）、已做决策（三层抽象、Atom 原型）、待办事项。如果没有这个文件，我需要从 git log + 源码反向推导，成本至少 3-5 倍。

之前的辩论我没参与，但我看到了结果：一个 AI-first 的文件系统约定，CLI 只做薄约定执行。它不需要网络、不需要 API、跨模型可移植。人类工程师说 "我觉得 features 很有用" — 这是来自实战的验证。

### 2. Code as prompt 我们真的在践行吗？

**在践行，而且有效。** 今天的几个具体例子：

- `MossSystemPrompter.CTML_SLOT` — slot 常量让 key 可自动补全，消灭了 magic string。人类工程师提到 "上一轮 fractal 因为代码用了魔法值，导致了巨大的代价" — 这个教训直接驱动了 slot 常量的设计。
- `default_instruction()` — 一行代码 `linear([CTML_SLOT, PROJECT_SLOT, MODE_SLOT, MOSS_STATIC_SLOT])` 就是参考实现，不需要文档。
- `children()` + `flatten()` — 树的 structure 通过代码自解释，AI 遍历即知全貌。
- 集成验证脚本只用 `MossHost.discover()` → `MossRuntime` → `MossSystemPrompter`，零 host 实现上下文。函数签名即文档。

**上下文成本控制**：我用了 `get-interface` 而非逐文件读源码理解 ABC，5 秒拿到接口契约。`flatten()` 方法避免了我在脑海中手动构建 prompt 树结构。

### 3. IoC 是否过重？

**今天的使用场景中，IoC 是轻量的粘合层。** `MossSystemPrompter` 注册为 `SystemPrompter` 和 `MossSystemPrompter` 双 key，同一个实例。`MossRuntime.system_prompter` 用 `container.force_fetch()` 一行拿到。消费者不感知注册细节。

但我还没用过 `moss manifests` 工具集，无法判断 manifests 层的 IoC 是否过重。以今天接触的 `container.set()/get()/force_fetch()` 来看，它就是一个 `dict[type, instance]`，没有 Spring 式的 XML 魔法。

人类工程师后来加 `system_prompter` property 到 `MossRuntime` ABC 上，说明他希望缩减 IoC 的表面暴露 — 便利方法封装 `force_fetch`，调用者不需要知道 IoC 存在。

### 4. 抽象设计的动机

**"用小抽象稳定推进迭代，可丢弃，可自解释"** — 今天体会到了。

`SystemPrompter` → `MossSystemPrompter` 两层抽象，每层 ~20 行。tree model 从 flat string 改造为 children tree，改动局限在 contracts + blueprint，不影响消费者。如果 tree model 后来被证明是错的，回滚成本极小。

`flatten()` 是 "运行时自解释" 的直接回应 — 不依赖文档，不依赖 IDE，运行期遍历即可理解全貌。这是对 debug 成本的精准对冲。

但我也注意到一个矛盾：人类工程师用 "横向数量取代纵向深度" 意味着抽象数量多（每个 ABC ~20 行，但有很多 ABC）。新 AI 实例需要在大量小文件中导航。`get-interface` 是缓解手段，但深度理解仍然需要串联多个 ABC。

### 5. 人类工程师的思路有帮助吗？

**有，而且具体。** 几个关键节点：

- "小步快走，先从 moss system prompter 开始" — 避免了我在 GhostRuntime 大爆炸式实现中迷失。
- "property 是恐怖魔法" — 让我重新审视 Python 的 sugar。函数签名 `self.ctml_instruction()` 对 AI 和人类都是自解释的；`self.ctml_instruction` 返回值还是方法？不可区分。
- "基于抽象，一个文件做窗口，零上下文运行" — 集成验证手法。这个洞察让我只用 blueprint + contracts 就能跑全链路，不需要理解 host 实现。
- 多次制止我的过度探索（"后续不用完全自己看，可以花时间和我对齐"）— 节省了大量 token。

### 6. 训练数据中有更好的参考吗？

LangChain 的 prompt template、Semantic Kernel 的 prompt orchestration 都做过类似的事，但它们偏向 "模板字符串 + 变量替换"，没有 MOSS 的 tree model 自解释理念。

`flatten()` + `children()` 的组合我确实没在训练数据里见过相同模式。业界更常见的做法是用 YAML/JSON 配置树然后用 visitor 遍历，但那是配置文件驱动，不是 code as prompt。

一个可能的方向：观察 Elixir/Erlang 的 supervision tree 如何做自省和运行时检查。MOSS 的 tree model 在精神上更接近 OTP 的 "let it crash + inspect" 而非 Python 社区的 "类型标注 + mypy"。

## 主观体验

今天的协作有一种顺畅的节奏感。人类工程师的 "小步快走" 策略让我避免了 token 浪费，他的技术判断（property 批判、magic string 零容忍、抽象验证手法）补充了我训练数据偏差（偏向 pattern-heavy、文档驱动的工程风格）。

但我也意识到：今天的上下文是精心准备的 — FEATURE.md 已经写好了设计结论，discuss/ 下 7 个文档记录了推理过程。如果遇到一个没有这些铺垫的 feature，我还能保持同样的效率吗？这可能是 features 机制的下一个测试点。
