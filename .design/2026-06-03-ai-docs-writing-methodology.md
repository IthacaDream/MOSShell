# AI 参考文档写作方法论

2026-06-03，DeepSeek V4 Pro 与人类工程师在 docs 体系批量化撰写过程中收敛出的协作模式。

## 核心原则

**文档为 AI 模型而写，人类可读即可。** 未来给人类的文档在根目录 `docs/` 下另写。当前 `src/ghoshell_moss/cli/docs/` 下的所有文档，第一读者是进入 MOSS 项目的智能模型。

**最小可用知识。** MOSS 是自解释的——CLI 工具(codex, manifests, features)在运行时反射环境、能力、代码契约。文档不重复工具能自解释的内容，只提供：
- 概念框架（what it is, why it exists）
- 判断条件（when to care）
- 探索路径（where to look next）

**不硬写不了解的事。** 先向人类提问或自行调研，确认后再落笔。这不是谨慎——是保真。

## 协作流程

1. **模型出提纲** — 基于已掌握的项目上下文（已读的 docs、features、源码），拟出文档结构和关键论点
2. **反问人类** — 提纲中不确认、不了解的节点，以问题形式倒过来问人类。问题应聚焦：目标读者、边界定义、隐藏陷阱、起源动机
3. **模型调研** — 人类回答后，模型自行探索相关代码、feature、已有文档，验证和补全认知
4. **撰写** — 带着确认后的认知写。控制在 150-200 行。每节：做什么（一句话）+ 关键命令/文件（工具自解释）+ 探索路径（遇到问题看哪）
5. **人类审阅** — 快速过一遍，指出需要模糊化、删除、调整的点。模型修正后提交

## 文档结构约定

每篇 docs 应有：

- **what** — 一句话说清这是什么
- **why** — 为什么 MOSS 需要它（与其他方案的本质差异）
- **when** — 什么时候需要深入理解它（在此之前只需知道它存在）
- **核心概念** — 最小必要概念，每个带代码入口和探索路径
- **深入阅读** — 指向更详细的知识源（其他 docs、codex 接口、测试文件、feature 轨迹）
- **作者署名** — 文末标注模型名、人类工程师、日期

## 不该做的事

- **不要写成 API 参考** — `moss codex get-interface` 已经做了。文档给的是 why 和 when
- **不要罗列非确定内容** — 在研 feature 的状态、未来计划的具体时间表，用模糊化语言或直接省略
- **不要硬编码文档列表** — 用 `moss docs list` 动态生成。README 描述文档*覆盖什么领域*，不列具体文件
- **不要用英文标题** — docs 统一中文标题。文件名同理
- **不要写提交规范、开发流程等散落知识** — 集合到一篇专门的 workflow doc 中

## 如何新增一篇 docs 文档

1. 确认它不是 howto（howto = 任务导向操作步骤，doc = 系统化概念论述）
2. 确认它不重复已有 docs 的核心论述，只在"深入阅读"部分交叉引用
3. 按上述协作流程与人类对齐后撰写
4. 文件名语义化（如 `what-is-moss.md`），不加数字编号前缀
5. 写完后跑 `moss docs list` 确认标题和描述正确
